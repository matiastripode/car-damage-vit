\begin{abstract}
Este trabajo presenta el desarrollo de un pipeline integral para la clasificación automática de daños en vehículos a partir de imágenes, basado en modelos de Vision Transformers (ViT). Se exploran arquitecturas ligeras y eficientes, junto con una estrategia de procesamiento mediante parches centrados en regiones de interés, con el objetivo de mejorar la precisión en escenarios reales.

El sistema implementado incluye componentes de inferencia y despliegue que abarcan una API REST, una interfaz web, una aplicación móvil y un mecanismo de retroalimentación humana (RLHF), permitiendo la mejora continua del modelo. Asimismo, el ciclo de vida del modelo es gestionado mediante MLflow, facilitando el versionado, monitoreo y reproducibilidad de los experimentos. La solución se integra en una arquitectura modular y escalable utilizando contenedores.

Asimismo, se analizan alternativas avanzadas de segmentación de instancias para enriquecer la detección de regiones relevantes, evaluando su potencial como complemento del pipeline de clasificación.
\end{abstract}

\section{Objetivo del Proyecto}
% =============================================================

El objetivo principal de este proyecto es construir un sistema capaz de \textbf{identificar
y clasificar automáticamente el tipo de daño presente en un vehículo} a partir de una
imagen, con el fin de acelerar los procesos de peritaje y estimación de reparaciones. Se considera dentro del alacen que el modelo se ejecute en un servidor de inferencia, para en una segunda etapa desplegarlo de forma local en el movil, eliminando la necesidad de conexión a la API para inferencia.

\subsection*{Objetivos específicos}
\begin{itemize}
  \item Implementar una arquitectura completa de Vision Transformer ligero: \\
        \texttt{apple/MobileViT-small} y \texttt{facebook/deit-tiny-patch16-224}, adecuadas para despliegue en
        dispositivos de recursos limitados.
  \item Desarrollar una estrategia de extracción de parches centrados en bounding boxes
        (224$\times$224 px) que convierta la detección de objetos en un problema de
        clasificación de parches.
  \item Construir una API REST de producción con soporte para retroalimentación humana
        (\textit{human-in-the-loop}) y persistencia de correcciones en MinIO.
  \item Proveer interfaces de usuario accesibles: aplicación móvil iOS y dashboard web.
  \item Implementar un pipeline de entrenamiento reproducible con MLflow para tracking y
        registro de modelos, facilitando la iteración y mejora continua.
  \item Conexión segura HTTPs con Traefik como reverse proxy, garantizando la integridad y confidencialidad de los datos.
  \item Evaluar una arquitectura mas compleja de segmentación de instancias (Mask2Former) con backbone SWIM \texttt{facebook/mask2former-swin-small-coco-instance} como mejora futura, para el uso de forma centralizada.
\end{itemize}

\subsection{Flujo de inferencia}

El flujo principal de una predicción sigue estos pasos:

Para IOS:
\begin{enumerate}
  \item El usuario captura o selecciona una imagen desde la app iOS.
  \item La imagen se envía vía HTTPS al endpoint \texttt{POST /predecir}.
  \item Traefik termina TLS y reenvía la solicitud a la API FastAPI.
  \item La API extrae la región de interés (ROI 224$\times$224 px), aplica el
        procesador de imágenes y realiza inferencia con MobileViT-small.
  \item Se devuelve la clase predicha, confianza y top-3 en formato JSON.
  \item Si RLHF está habilitado, la ROI y la predicción se persisten en MinIO
        (de forma asíncrona, sin bloquear la respuesta).
\end{enumerate}
Para Web UI, el proceso es similar, la diferencia radica en que el usuario sube una imagen a la interfaz y selecciona una región de interés para el análisis.

\subsection{Preprocesamiento del dataset}

La estrategia central del proyecto convierte el problema de detección de objetos en
\textbf{clasificación de parches}:

\begin{itemize}
  \item Por cada anotación, se extrae un parche de 224$\times$224 px centrado en el
        bounding box (\texttt{CarDamageDataset}, \texttt{src/vit/data/dataset.py}).
  \item Se generan parches de fondo (\texttt{fondo}) con IoU = 0 respecto a cualquier
        anotación, en proporción 1:1 con los parches de daño.
  \item El desbalance de clases (tire\_flat tiene 11,6$\times$ menos instancias que
        scratch) se mitiga con \texttt{WeightedRandomSampler} en el DataLoader.
\end{itemize}

\subsection{MobileViT-small: Arquitectura híbrida CNN + Transformer}

Según el paper original de MobileViT (\texttt{arXiv:2110.02178}), el objetivo es
combinar la \textbf{eficiencia local} de las CNN con la \textbf{modelación global}
de los Transformers en un backbone apto para dispositivos móviles.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.8\textwidth]{Diagrama_MobileViT.png}
  \caption{MobileViT enfatiza el uso de ViTs ligeros, de baja latencia, y precisos para aplicaciones en dispositivos con recursos restringidos.}
\end{figure}


\textbf{Idea central del bloque MobileViT (``transformers as convolutions''):}
\begin{enumerate}
  \item Se extraen \textbf{representaciones locales} con convoluciones
        (conv $n\times n$ + conv $1\times1$).
  \item El mapa se ``\textit{unfold}'' en parches pequeños, se procesa con
        bloques Transformer para capturar \textbf{contexto global}, y luego se
        ``\textit{fold}'' nuevamente al espacio espacial.
  \item Se aplica una etapa de \textbf{fusión} con convoluciones para mezclar
        información local + global en la salida del bloque.
\end{enumerate}

Esta formulación reemplaza el procesamiento local de una convolución estándar por
procesamiento global con autoatención, manteniendo un diseño liviano y práctico
para inferencia móvil.

\textbf{Ventajas reportadas en el paper:}
\begin{itemize}
  \item \textbf{Mejor compromiso precisión/parámetros:} en ImageNet-1k, MobileViT
        reporta 78,4\% top-1 con $\sim$6M de parámetros, superando en ese rango
        a MobileNetV3 (+3,2 pp) y DeiT (+6,2 pp).
  \item \textbf{Backbone generalista:} también muestra mejoras en tareas densas
        (detección/segmentación), no solo en clasificación.
  \item \textbf{Enfoque mobile-first:} el trabajo enfatiza que optimizar solo FLOPs
        no garantiza baja latencia real en móvil; por eso prioriza diseño liviano,
        latencia y facilidad de integración en pipelines de visión.
\end{itemize}

Ambos modelos se cargan desde HuggingFace con \texttt{AutoModelForImageClassification},
reemplazando la cabeza de clasificación por una capa lineal de 7 salidas \\
(\texttt{src/vit/models/factory.py}):

\noindent\begin{minipage}{\linewidth}
\begin{lstlisting}[language=Python, caption={Carga del modelo desde HuggingFace.}]
def cargar_modelo(nombre, num_clases, attn_implementation=None):
    procesador = AutoImageProcessor.from_pretrained(nombre, use_fast=True)
    modelo = AutoModelForImageClassification.from_pretrained(
        nombre,
        num_labels=num_clases,
        ignore_mismatched_sizes=True,
        attn_implementation=attn_implementation,
    )
    return modelo, procesador
\end{lstlisting}
\end{minipage}

\subsubsection{Procesamiento de imágenes en entrenamiento:}
\begin{itemize}
  \item \textbf{Entrada del pipeline:} orquesta \texttt{descargar\_dataset.py},
        \texttt{exportar\_anotaciones.py} y luego \texttt{entrenar.py}.
  \item \textbf{Conversión a COCO:} \texttt{exportar\_anotaciones.py} convierte los
        \texttt{bbox} normalizados de FiftyOne a coordenadas absolutas en píxeles
        para \texttt{train/validation/test.json}.
  \item \textbf{Estrategia de parcheo (224$\times$224):} en
        \texttt{CarDamageDataset} se genera un parche positivo centrado por cada
        anotación de daño y, además, parches de \texttt{fondo} con
        \texttt{ratio\_fondo=1.0} (configurable) evitando solapamiento relevante con
        daños (IoU $< 0.1$).
  \item \textbf{Transforms de entrenamiento:}
        \texttt{RandomResizedCrop(224, scale=(0.8,1.0))},
        \texttt{RandomHorizontalFlip}, \texttt{RandomRotation(15)},
        \texttt{ColorJitter}, \texttt{Normalize(mean/std del processor)}, \texttt{ToTensor}
        y \texttt{RandomErasing}. Estas transformaciones ayudan a:
        \begin{enumerate}
          \item Mejorar robustez a cambios de escala/orientación/iluminación (\texttt{crop}, \texttt{flip}, \texttt{rotation}, \texttt{color jitter})
          \item Reducir sobreajuste a texturas puntuales y oclusiones parciales (\texttt{random erasing})
          \item Estabilizar la entrada al backbone preentrenado (\texttt{normalize})
        \end{enumerate}
          En relación con el EDA, esto es especialmente útil para clases difíciles
          con alta proporción de objetos pequeños (como \texttt{crack}) y para daños
          cercanos a bordes, donde pequeñas variaciones geométricas pueden cambiar
          la evidencia visual disponible en el parche.
        
  \item \textbf{Transforms de validación/test:} solo \texttt{ToTensor} +
        \texttt{Normalize}, sin augmentaciones estocásticas.
  \item \textbf{Muestreo balanceado:} el \texttt{DataLoader} de train usa
        \texttt{WeightedRandomSampler} con pesos inversos a la frecuencia de clase para
        mitigar el desbalance observado en EDA.
\end{itemize}

\subsubsection{Flujo operativo del pipeline de entrenamiento(8 etapas):}
\begin{enumerate}
  \item \textbf{Parseo de argumentos de entrada:} \texttt{--config} (requerido),
        \texttt{--env} (\texttt{dev|staging|prod}) y flags opcionales de MLflow
        (\texttt{--mlflow-uri}, \texttt{--mlflow-train-experiment}, \\
        \texttt{--mlflow-eval-experiment}, \texttt{--mlflow-register-name}).
  \item \textbf{Resolución de configuración:} la ruta YAML se normaliza contra la raíz
        del repositorio cuando se pasa como relativa; si es absoluta, se usa sin cambios.
  \item \textbf{Preparación idempotente de datos:}
        si faltan splits en \texttt{data/raw/\{train,validation,test\}} (con
        \texttt{state.json} y \texttt{dataset\_info.json}), se ejecuta
        \texttt{scripts/descargar\_dataset.py}; si faltan anotaciones COCO
        (\texttt{train.json}, \texttt{validation.json}, \texttt{test.json}),
        se ejecuta \texttt{scripts/exportar\_anotaciones.py}.
  \item \textbf{Validación temprana de entorno MLflow:}
        cuando se informa \texttt{--mlflow-uri}, se verifica que el intérprete local
        tenga el módulo \texttt{mlflow} instalado antes de iniciar entrenamiento.
  \item \textbf{Entrenamiento:}
        se invoca \texttt{scripts/entrenar.py} propagando \texttt{config} y \texttt{env},
        y opcionalmente parámetros de tracking/registro en MLflow. \\
        En este paso, \texttt{src/vit/train/trainer.py} aporta
        \textbf{early stopping}, \textbf{checkpointing} del mejor modelo y
        \textbf{logging} de métricas/artefactos en MLflow.
  \item \textbf{Resolución y verificación de checkpoint esperado:}
        se exige la presencia de \texttt{best\_model.pt} en
        \texttt{checkpoints/<modelo\_slug>/best\_model.pt}, o en el \texttt{output\_dir}
        definido en la configuración si fue sobrescrito.
  \item \textbf{Evaluación sobre test:}
        se ejecuta \texttt{scripts/evaluar.py} con el checkpoint validado en el paso
        anterior, reutilizando \texttt{config}/\texttt{env} y, si corresponde, logging
        en experimento de evaluación de MLflow.
  \item \textbf{Cierre exitoso:}
        el pipeline finaliza reportando ``\texttt{Pipeline completado.}''.
\end{enumerate}

\textbf{Comportamiento ante fallos:}
\begin{itemize}
  \item El pipeline corta la ejecución si falla cualquier subproceso invocado
        (\texttt{subprocess.run(..., check=True)}).
  \item Si tras entrenamiento no existe el checkpoint esperado, se aborta con
        \texttt{FileNotFoundError} para evitar una evaluación inconsistente.
\end{itemize}

\textbf{Contrato entre etapas (train $\rightarrow$ eval):}
\begin{itemize}
  \item Artefacto obligatorio: \texttt{best\_model.pt}.
  \item Ubicación por defecto:
        \texttt{checkpoints/<modelo\_slug>/best\_model.pt}.
  \item Ubicación alternativa: \texttt{output\_dir} del YAML de modelo.
\end{itemize}

        unificando el flujo RLHF sin necesidad de la app iOS.
  \item \textbf{Ancho máximo fijo (1024 px):} se aplica CSS personalizado para mejorar
        la legibilidad en pantallas anchas.
  \item \textbf{Gestión de estado por sesión:} se usa \texttt{st.session\_state} para
        preservar predicción y feedback entre rerenders, evitando pérdida de resultados
        al interactuar con la UI.
\end{itemize}

\subsection{Infraestructura RLHF}

Cuando el usuario corrige una predicción errónea (desde la app iOS o la UI web):
\begin{enumerate}
  \item Se envía \texttt{roi\_key} (clave del parche en MinIO) y \texttt{clase\_correcta}.
  \item La API genera una anotación en formato YOLO y la guarda junto al parche ROI
        en el bucket \texttt{rlhf} de MinIO.
  \item Estos datos quedan persistidos en MinIO y disponibles para un eventual pipeline de re-entrenamiento (fuera del alcance del presente proyecto).
\end{enumerate}

\section{Conclusiones y Mejoras Futuras}

\subsection{Conclusiones}

\begin{enumerate}
  \item \textbf{MobileViT-small es la arquitectura ganadora.} Con solo 5,6M parámetros,
        logra 77,92\% de accuracy y macro F1 de 0,7596, superando a DeiT-tiny en ambas
        métricas. Su arquitectura híbrida CNN + ViT le permite capturar mejor las
        características locales de daños pequeños.

  \item \textbf{La estrategia de parches es efectiva.} Convertir detección en clasificación
        permite reutilizar ViTs pre-entrenados en ImageNet sin necesidad de fine-tuning
        de cabezas de detección más complejas.

  \item \textbf{El pipeline end-to-end es funcional y desplegable.} Desde la captura
        móvil hasta la retroalimentación humana, el sistema opera de forma integrada
        con un único \texttt{docker compose up}.

  \item \textbf{La recolección de feedback humano está operativa.} Las correcciones
        ingresadas desde la app iOS o la UI web se persisten en MinIO como pares
        ROI + anotación YOLO. El pipeline de re-entrenamiento automático queda fuera
        del alcance de este proyecto y se propone como trabajo futuro.
\end{enumerate}


\subsection{Mejoras futuras}

\begin{itemize}
  \item Ejecución del modelo MobileViT directamente en el mobile.
  \item Servidor de inferencia y WebUI para la ejecución de Mask2Former, con el objetivo de obtener segmentación de instancias y máscaras precisas que permitan delimitar el contorno del daño, información valiosa para estimaciones de reparación en imagenes de campo amplio.

\subsubsection{Detección y segmentación de instancias con SWIN+Mask2Former}

Basado en el paper de referencia \texttt{arXiv:2112.01527}
(\textit{Masked-attention Mask Transformer for Universal Image Segmentation}),
Mask2Former propone una arquitectura \textbf{unificada} para segmentación semántica,
de instancias y panóptica sobre una misma base de \textit{mask classification}.

\textbf{Componentes principales de la arquitectura:}
\begin{itemize}
  \item \textbf{Backbone Swin Transformer \texttt{ICCV 2021, Swin Transformer}:} la imagen se divide en parches y se
        procesa en una arquitectura jerárquica por etapas, donde cada etapa reduce la
        resolución y aumenta canales mediante \textit{patch merging} (análogo a una
        pirámide de features). Dentro de cada etapa, la atención se calcula en ventanas
        locales no superpuestas (\textit{W-MSA}) y en la capa siguiente se desplazan las
        ventanas (\textit{SW-MSA}) para habilitar interacción entre ventanas vecinas.
        Esta combinación preserva contexto local y conexión global progresiva con costo
        computacional aproximadamente lineal respecto al tamaño de imagen, lo que lo
        vuelve especialmente útil como backbone para tareas densas (detección/segmentación).
        \begin{figure}[H]
          \centering
          \includegraphics[width=0.4\textwidth]{swin_transformer.png}
          \caption{Mecanismo de atención jerárquica basado en ventanas deslizantes.}
        \end{figure}
        
  \item \textbf{Pixel decoder:} fusiona y refina features de múltiples resoluciones
        para producir embeddings por píxel de mayor calidad.
  \item \textbf{Transformer decoder con \textit{masked attention}:} cada \textit{query}
        atiende principalmente dentro de su región de máscara predicha, en lugar de
        atender toda la imagen como en la \textit{cross-attention} estándar.
        Esto mejora convergencia y precisión, especialmente en segmentación de instancias.
\end{itemize}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.5\textwidth]{Diagrama_Mask2Former_swin.png}
  \caption{Arquitectura de Mask2Former-Swin.}
\end{figure}

\textbf{Idea clave de Mask2Former (arXiv:2112.01527):}
la atención enmascarada y el uso eficiente de features de alta resolución/multiescala
permiten una segmentación más precisa (incluyendo objetos pequeños) con una receta
de entrenamiento más estable que enfoques universales previos.

\textbf{Métricas de medición para CarDD (COCO)}

Estas métricas se calculan sobre dos tareas relacionadas:
\begin{itemize}
  \item \texttt{bbox\_*}: calidad de \textbf{detección} (localización con cajas).
  \item \texttt{segm\_*}: calidad de \textbf{segmentación de instancia} (máscara por daño).
\end{itemize}

\textbf{Métricas de detección (\texttt{bbox})}
\begin{itemize}
  \item \texttt{bbox\_mAP}: promedio de AP (\textit{Average Precision}) sobre múltiples
        umbrales IoU (0.50:0.95). Es la métrica más estricta y representativa del
        rendimiento general. En CarDD indica qué tan bien el modelo localiza distintos
        tipos de daño (desde raspones pequeños hasta daños grandes).
  \item \texttt{bbox\_mAP50}: AP con IoU=0.50. Es más permisiva que
        \texttt{bbox\_mAP}, por eso suele ser más alta. En CarDD sirve para verificar
        si el modelo ``encuentra'' daños de forma general, aunque la caja no sea
        muy precisa.
  \item \texttt{bbox\_mAR100}: \textit{recall} promedio permitiendo hasta 100
        predicciones por imagen. En CarDD es útil porque puede haber múltiples daños
        por imagen; refleja cuántos daños reales logra recuperar el modelo.
\end{itemize}

\textbf{Métricas de segmentación (\texttt{segm})}
\begin{itemize}
  \item \texttt{segm\_mAP}: promedio de AP sobre IoU 0.50:0.95 usando máscaras.
        Mide calidad global de segmentación y es exigente en el contorno. En CarDD es
        clave porque hay clases finas (ej. \texttt{crack}, \texttt{scratch}) donde la
        forma del daño importa.
  \item \texttt{segm\_mAP50}: AP de segmentación con IoU=0.50. Es más tolerante al
        detalle fino del borde. En CarDD ayuda a evaluar si la máscara cubre la región
        correcta, aunque no sea perfecta en contornos.
  \item \texttt{segm\_mAR100}: \textit{recall} promedio en segmentación con hasta 100
        predicciones por imagen. En CarDD complementa a \texttt{segm\_mAP}, mostrando
        si el modelo se está ``perdiendo'' daños en escenas con varias instancias.
\end{itemize}

\textbf{Lectura práctica para este proyecto}
\begin{itemize}
  \item Si \texttt{bbox\_*} $>$ \texttt{segm\_*} por mucho margen: el modelo detecta daños,
        pero falla en delinear bien la máscara.
  \item Si \texttt{mAP50} es alto pero \texttt{mAP} cae: hay aciertos gruesos, pero falta
        precisión fina (común en daños pequeños o finos).
  \item Si \texttt{mAR100} es bajo: el modelo está dejando daños sin detectar, incluso
        permitiendo muchas predicciones por imagen.
\end{itemize}

Los resultados demuestran que la arquitectura de segmentación de instancias es un
camino de mejora concreto y prometedor:

\begin{itemize}
  \item El modelo fue \textbf{ajustado (fine-tuning) sobre CarDD} durante 20 épocas.
        En este contexto, un mAP@0.50 de \textbf{0,706} en detección bbox y
        \textbf{0,722} en segmentación de instancias confirma un desempeño sólido para
        el dominio automotriz.
  \item En evaluación cuantitativa de segmentación, se obtuvieron
        \texttt{segm\_mAP = 0,548}, \texttt{segm\_mAP50 = 0,722} y
        \texttt{segm\_mAR100 = 0,623}, resultados consistentes con una buena capacidad
        de recuperación de instancias.
  \item La segmentación de instancias permite delimitar con precisión el contorno del
        daño, información valiosa para estimaciones de reparación.
  \item En pruebas cualitativas con imágenes externas al dataset, el modelo mantuvo
        detecciones plausibles, lo que sugiere una generalización útil fuera del set de
        entrenamiento.
  \item A diferencia de la estrategia de parches actual, Mask2Former procesa la imagen
        completa en una sola pasada, eliminando la necesidad de extraer ROIs manualmente.
  \item El desempeño en objetos pequeños (AP$_\text{small}$ = 0,051) indica que el
        modelo aún tiene margen de mejora, especialmente con augmentación específica para
        objetos pequeños y mayor resolución de entrada.
\end{itemize}

La integración de Mask2Former como backend de inferencia de la API representa la
\textbf{dirección de evolución prioritaria} del sistema centralizado.
